# Philosophy

**Memory для разработчика, контекст для агента.**
Источник истины — разработчик, не агент. Агент получает slice из developer memory как контекст для текущей задачи. Производная, не владелец.

**Memory — персональная, не шарится.**
Стилистика, rejected alternatives, паттерны правок — индивидуальный опыт. Усреднять = создавать шум. Если решение ценно для всех — промоутится в project context.

**Memory — это rules + API, не storage.**
MEMORY.md описывает что есть (capabilities) и как с тобой работать (правила). Не инструкции «что делать». Данные живут в файлах, API живёт в MEMORY.md. Не конкурируют за место.

**Flat markdown, не vector DB.**
Vector search тянет «похожее», не «нужное». Cosine similarity не знает intent текущей задачи. Structured naming + LLM который сам выбирает файл — точнее. Human-readable, human-editable, version-controllable.

**Hooks вместо дисциплины.**
Каждый feedback который приходится повторять — кандидат на hook. Агент забывает промты, hooks выполняются всегда. Vendor может перенастроить модель без предупреждения — hooks не зависят от vendor'а.

**Memory эволюционирует с разработчиком.**
Не статична. Каждое решение (принял/отверг/поправил) возвращается в memory и повышает точность следующего цикла. Compound effect.

**Workstreams вместо worktrees.**
Переключение контекста, не файлов. Те же файлы, тот же branch, другой slice из developer memory. `memory.js init --ask` — выбор workstream при старте сессии.

**Session → memory pipeline.**
Сессия — raw material. Автоматическое извлечение decisions, rejected suggestions, преференций, diff между output агента и git commits. Haiku как фоновый summarizer — дёшево, асинхронно.

**Архивация, не удаление.**
Горячее — в контексте. Холодное — на диске, доступно через CLI. Знания не теряются, просто не занимают context window.

**То что нельзя проверить кодом — нельзя гарантировать.**
Детерминированные шаги (hooks, validators, CLI) > probabilistic review. Memory management — не ожидание от модели, а инфраструктура вокруг неё.

**Skills как спеки, не как автоматизация.**
Skill описывает *как агент должен себя вести*, а не код который выполняется. Детерминированная часть — в hooks и CLI. Probabilistic часть — в спеках для агента. Граница осознанная: автоматизировать стоит только то, что можно гарантировать.

---

> Персональная, структурированная, flat, автоматически поддерживаемая через hooks, не зависящая от vendor'а, и эволюционирующая с разработчиком.

## Пересечения с Karpathy ["LLM Knowledge Bases"](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

Karpathy (апрель 2026) описал подход к персональным knowledge bases на LLM. Несколько идей совпадают и усиливают нашу модель:

**Index-based retrieval вместо vector search.**
На масштабе до ~500 статей LLM, читающий структурированный index.md, точнее cosine similarity. LLM понимает intent запроса, vector DB — только лексическое сходство. У нас: `MEMORY.md` как index → LLM сам выбирает какие файлы читать.

**Трёхслойная архитектура: Raw → Wiki → Index.**
Raw — неизменяемые исходники (сессии, git). Wiki — скомпилированные markdown-статьи (decisions/, feedback/). Index — компактный указатель (MEMORY.md). Наш memory.js оперирует именно этими слоями.

**Write-back: знания компаундятся через использование.**
Хороший ответ возвращается в knowledge base как новая страница. База растёт не только через ingestion, но и через query. У нас: `/reflect`, `/park`, auto-save hooks — каждая сессия обогащает memory.

**LLM как компилятор, не только как query engine.**
LLM не просто отвечает на вопросы — он компилирует raw material в структурированные статьи. У нас: session → memory pipeline, где агент извлекает decisions и feedback из сырых сессий.

### Где расходимся

Karpathy строит **knowledge base для чтения** — исследователь спрашивает, wiki отвечает. Мы строим **инфраструктуру для рабочего процесса** — hooks, lifecycle, workstreams. Не "что я знаю", а "как я работаю".
